## 1. The stack as it stands today

### Hardware

NAS box ("depot") running **Arch Linux + ZFS**. Bringup/management
lives in a Ruby tool the user wrote called `depot`, at
`~/code/depot/`. Re-running `depot install` is idempotent and
brings every service to current desired state. `depot update aviary`
is the one-off that pulls + rebuilds + migrates just the aviary
container without touching anything else.

### Services running on depot

All in docker-compose, persisted under `~/hdds/.config/<service>/`:

- **Jellyfin** — media server. Source of truth for users, libraries,
  user images (avatars), playback state, HLS streams.
- **Sonarr / Radarr / Prowlarr** — show/movie acquisition stack
- **Jellyseerr** — discovery layer (TMDB metadata + trending lists)
- **qBittorrent / Gluetun / Sabnzbd / Unpackerr** — download path
- **Aviary** — Phoenix LiveView app at `~/code/aviary/`
- **Cloudflare Tunnel (cloudflared)** — public HTTPS at
  `pachulski.tv`. Replaces an earlier Tailscale Funnel setup; tunnel
  token + hostname persisted under `~/hdds/.config/cloudflared/`.
- **Tailscale** — internal access (tailnet URL: depot.tail<...>.ts.net)

Service code at `~/code/depot/services/<name>/<name>.rb` — each
exposes `install`, `update`, `summary`, `install_prompt`.

### Aviary — the household media UI

Phoenix 1.7 / LiveView 1.1 / Elixir 1.18, Tailwind v4. Data store:
SQLite at `/data/aviary.db` (container-mounted volume on the NAS).

**Brand / design language** (this matters for tvOS too — should
carry the same aesthetic):

- Vibe: "editorial-museum-cabinet" — less chrome, museum-object-label
  feel, single oxblood accent on warm paper, never material/cyber
- Typography: **Newsreader** (display serif headlines) + **Fraunces**
  italic (body sub-copy, captions) + **Instrument Sans** (tracked
  uppercase eyebrows/labels/UI text)
- Single accent color: oxblood (deep red)
- Configurable brand name via `TAB_TITLE` env var (currently
  "Pachulski TV" for the user's family deploy)
- Tab titles: `<page> · Pachulski TV` — page-specific part set per
  LiveView, suffix appended in root.html.heex
- Light/dark themes via `[data-theme=day|night]` on `<html>`,
  defaults to night, persisted in localStorage

**Buttons / vocabulary conventions** (per the user's saved memory):

- Play CTAs are always **"Watch"**, never "Request" or "Play". For
  shows that aren't downloaded yet, the big CTA is **"Add to
  library"** (acts as "save for later" — clicking it adds to the
  user's library_entries AND triggers Sonarr/Radarr if needed; does
  NOT auto-play)
- Downloads are invisible to users (no sysadmin framing in copy)
- Confirmed sections in the IA: **Shows**, **Movies** (don't presume
  Jellyfin/Jellyseerr terms beyond this without asking)
- The detail page subtitle under the CTA is "Watchable now" for
  already-downloaded items, or a time-estimate ("Roughly N minutes
  until watchable") / live progress for not-yet-downloaded items —
  same big CTA in both cases (consistent UI regardless of
  download state, per the user's explicit ask)

**Top-level IA** — five sections, mobile bottom tab bar + desktop
masthead:

- **Home** — Continue Watching → Upcoming → Family Recommended (in
  that order, lowest priority last). Tab visible if ANY section has
  content; hidden otherwise.
- **Library** — per-user `library_entries` filtered by Shows/Movies
  toggle. Library tab visible if user has any entries.
- **Discover** — Jellyseerr trending across configured services
- **Search** — Jellyseerr `/search/multi`, debounced, recent searches
  surfaced in empty state
- **Settings** — YOU (signed in as / sign out) / DISPLAY (theme) /
  YOUR USAGE / TOTAL STORAGE / **Home screen** (PWA install steps,
  iOS-Safari-only)

### Key modules in aviary

- `Aviary.Jellyfin` — API client. `list_users`, `audio_streams`,
  `fetch_poster`, `fetch_user_image`, `hls_url`, `resume_items`,
  `next_up_across_library`, etc. Auth via per-user `x-emby-token`.
- `Aviary.Catalog` — `get_show/2`, `get_movie/2`, `list_shows/1`,
  `list_movies/1` — return library OR discover shape depending on
  whether Jellyfin has the item; per-user library_entries gate the
  list endpoints
- `Aviary.Library` — `member?/2`, `add/2`, `remove/3` — per-user
  library curation (sqlite `library_entries` table)
- `Aviary.Home` — `continue_watching/1` — assembles + dedupes
  resume/next-up/recently-watched/latest into a single marquee feed
- `Aviary.Upcoming` — Jellyseerr-driven release schedule per series
  the user has touched
- `Aviary.Search`, `Aviary.RecentSearches`, `Aviary.Discover`
- `Aviary.Recommendations` — the recent big feature: family
  recommendations with per-item dismissal. Two tables:
  `recommendations(from_user_id, to_user_id, tmdb_id, kind)` and
  `dismissed_recommendations(user_id, tmdb_id, kind)`. Active recs
  = exists in recommendations + not in dismissed_recommendations.
- `Aviary.Nav` — `visibility/1` decides which top-level tabs render
  (Discover/Search always; Home/Library conditional on content)
- `Aviary.LocalTime` — timezone-aware date helpers
- `Aviary.WatchProgress` — Sonarr/Radarr queue.timeleft parsing →
  "Roughly N minutes until watchable" copy

### Image proxy

`AviaryWeb.ImageController` proxies:
- `/image/:item_id?kind=backdrop|primary` → Jellyfin item images
  (auth-required, served with aggressive `immutable` cache headers)
- `/image/tmdb/:size/*path` → TMDB images
- `/user-image/:user_id?tag=<PrimaryImageTag>` → Jellyfin user
  avatars (semi-public, no auth needed)

### Auth shape

`Aviary.Auth.log_in/2` POSTs to Jellyfin's `/Users/AuthenticateByName`.
Returns `{:ok, %{id, username, token, primary_image_tag}}`. Token is
stored in the session and sent as `x-emby-token` on every backend
call from then on.

### Streaming

HLS via Jellyfin's `/Videos/{id}/master.m3u8`. Aviary builds the URL
with `Aviary.Jellyfin.hls_url/3`. Currently capped at stereo (2ch
AAC) — an earlier surround experiment broke playback and was
reverted; the comment block on `hls_url` warns about how delicate
Jellyfin's playback-decision pipeline is.

### PWA install flow (iOS-only nudge, gentleman version)

- Root layout detects iOS Safari + not standalone synchronously in
  `<head>`, sets `data-needs-install="true"` on `<html>` before paint
- One-time toast on page load via `maybeShowInstallHint()` in
  `assets/js/app.js` — only fires if iOS + not standalone + no
  localStorage `aviary:install-hint-shown` flag yet
- Persistent Settings → "Home screen" section with the 5 install
  steps (CSS-gated by `data-needs-install`) — `Layouts.install_steps/1`
- The instructions match current Liquid Glass Safari: tap "…" menu
  bottom-right → Share → down-arrow circle → Add to Home Screen →
  open from home screen

## 3. tvOS client — conversation summary

User wants an Apple TV app — believes it's the make-or-break for
family adoption ("first class"). Has an Apple developer account,
ships iOS apps for work via **Expo + EAS** workflow. Has never
shipped a tvOS app.

### User's stated constraints / preferences

- **Expo, not bare RN, not Swift.** Specifically wants Expo because:
  - **CNG (Continuous Native Generation)** decouples code from native
    project config; the iOS/Android dirs are generated reproducibly
    from `app.config.js`, never edited by hand
  - **EAS Build** handles certificates / provisioning profiles
    automatically at build time in Expo's cloud
  - **Expo Updates (OTA)** ships JS changes without rebuilds — only
    native dep changes need a rebuild

  The user pushed back hard on an earlier framing where Claude
  treated "bare workflow" as a friction point — that framing was
  ~6 years stale. CNG IS the modern Expo workflow.

### What's confirmed possible via Expo docs
(`https://docs.expo.dev/guides/building-for-tv/`)

- **CNG for tvOS** — yes, via `@react-native-tvos/config-tv` plugin
  + `EXPO_TV=1` env var (or `isTV: true` plugin parameter). Run
  `npx expo prebuild --clean` to generate.
- **EAS Build for tvOS** — yes. Pattern from docs: `development_tv`
  / `preview_tv` profiles extend the phone profiles and just set
  `env: { EXPO_TV: "1" }`. Same source, two targets.
- **OTA via Expo Updates** — yes, channels work on tvOS (the docs'
  `eas.json` example shows `"channel"` configured on TV profiles).
- **SDK 56+** ships `react-native-tvos` as the RN dep automatically
  when EXPO_TV is set; upgrading SDK upgrades the TV repo with it.
  No manual version pinning.

### Known limitations (from the docs)

- DevClient on Apple TV is partial in SDK 54+: "basic operations
  with a local or tunneled packager are supported." Authentication
  to EAS and listing EAS builds/updates from inside the DevClient
  aren't there yet. **Ship path is fine; this is a dev-flow paper
  cut, not a blocker.**
- General: not every RN library has tvOS builds. Evaluate each.

### Backend reuse (the easy part)

Phoenix backend mostly already supports this:
- Almost all logic lives in context modules (`Aviary.Catalog`,
  `Aviary.Home`, `Aviary.Recommendations`, etc.) — none of them
  care whether the caller is LiveView or a JSON controller
- Auth flow is already token-based (Jellyfin tokens) — native client
  stores the token, sends as bearer header on every call
- HLS URLs work as-is — `AVPlayer` plays them natively (gets surround
  + AirPlay for free, unlike browser HLS which was unreliable enough
  that the surround experiment was reverted)
- LiveView's WebSocket can be skipped on tvOS; REST is sufficient
  for media browsing

**Backend work needed:** ~2-3 days of focused work. New
`AviaryWeb.API` namespace with JSON controllers wrapping existing
context calls. Endpoints needed for:
- Auth (POST credentials → token)
- Home (Continue Watching, Upcoming, Family Recommended)
- Library (Shows/Movies, filtered by user)
- Discover (per-service trending rows)
- Search
- Detail (show + episodes; movie)
- Watch (start playback → HLS URL)
- Recommendations (send, list household members, dismiss)

### App work needed

~2-3 weeks of focused evening work for a v1, per the conversation.
Reduced from the original 2-4 week estimate after the user pointed
out the Expo workflow handles most of the friction Claude had
priced in.

Real cost that stays: **focus engine design + Siri Remote
handling**. tvOS UX is genuinely different from iOS — `next/upFocus`
routing, focus visualizers, autoplay-on-focus behavior, swipe-up
metadata gestures. That's tvOS-specific design work the toolchain
doesn't help with.

### Bridge option mentioned but not adopted

**Swiftfin** (Jellyfin's open-source tvOS client) was mentioned as a
"ship to family tonight while you build" bridge. User did not
explicitly accept or reject; it's a no-cost option that gives them
basic Jellyfin-on-TV today but loses Pachulski TV branding + Aviary
curation + Family Recommendations + the "Watch" vocabulary. The
whole point of the Aviary tvOS build is the curation layer; Swiftfin
won't give the family that.

## 4. Suggested first move in the fresh chat

If continuing toward shipping tvOS:

1. **Sketch the JSON API surface.** Mirror the data shapes the
   existing LiveViews already assemble (HomeLive's mount, CatalogGrid
   items, marquee items). Each LiveView's mount/handle_event flow
   is the spec for the corresponding endpoint.
2. **Scaffold the Expo + react-native-tvos project** with the
   `@react-native-tvos/config-tv` plugin set up and an
   EAS Build profile for tvOS.
3. **Implement auth screen** end-to-end (sign in to Jellyfin via
   the new aviary JSON endpoint, store token, navigate to Home).
4. **Build one full screen** (Home) end-to-end before moving wide —
   forces real-shaped data + focus engine + Marquee-on-tvOS design
   decisions early, surfaces gotchas while scope is small.

### Things to consider while building

- Maintain the editorial aesthetic in tvOS. Same Newsreader/Fraunces/
  Instrument Sans typography (Google Fonts works on tvOS via standard
  RN font loading). Same oxblood accent. Same museum-cabinet vibe —
  resist generic Material/iOS chrome.
- No "Add to library" abbreviated copy — same vocabulary as web.
  Same Family Recommendations badge ("Chris thinks you'll like this").
- Reuse the same `recommendations` table; dismissal logic is
  identical. Same sender-side popover for the Recommend flow.
- Continue Watching and Upcoming are the top-priority signals; Family
  Recommended sits below them on Home (per the user's decision).
- The "Family Recommended" row hides items the recipient already has
  in their library — same rule on tvOS.
